# Space Modeling

Система моделирования движения малых спутников Земли. Текущий фокус проекта -
прозрачная и воспроизводимая пропагация орбиты с возможностью включать и
отключать отдельные физические возмущения.

## Статус

В активной части проекта сейчас используются:

- `dynamics/` - модель динамики, параметры среды и численное интегрирование;
- `visualization/` - 2D/3D визуализация орбит, трасс на карте, анимаций и
  эллипсоидов ковариации;
- `notebooks/` - демонстрационные ноутбуки для локального запуска и Colab;
- `history/` - сохраненные результаты предыдущих экспериментов.

Папки `estimation/` и `experiments/` удалены как неиспользуемые заготовки.

## Физическая модель

Состояние спутника задается в СИ как

```text
y = [r_x, r_y, r_z, v_x, v_y, v_z],
```

где `r` - геоцентрический радиус-вектор [м], `v` - скорость [м/с]. В модуле
`dynamics.propagator` интегрируется система

```text
dr/dt = v
dv/dt = a_total
a_total = a_earth + a_J2 + a_sun + a_moon + a_drag + a_srp
```

Каждая сила включается или отключается через `ForceModelConfig`.
Для воспроизводимых addition-study есть helper `make_recommended_force_config()`:
он включает body-fixed J2, NRLMSISE-00 и коническую тень SRP, но оставляет
ИК-давление Земли, релятивистскую поправку и приливы выключенными до отдельной
валидации их вклада.

## Проверка по precise-orbit данным

Основная проверка модели в ноутбуках выполняется не по SGP4/TLE, а по
precise-orbit продуктам, которые содержат готовый шестимерный вектор состояния
аппарата. Это важно для отладки: скорость берется из внешнего POD-продукта, а
не восстанавливается конечными разностями по координатам.

Основной список `VELOCITY_REFERENCE_SOURCES` включает:

- ESA Swarm POD `SP3xCOM_2_` для Swarm A/B/C: SP3c velocity records (`#dV` и
  строки `V`);
- CASSIOPE / Swarm-E `CAS_Orbit_GEO_...sp3.zip`: SP3 с 1 Hz позициями и
  скоростями для спутника `L63`;
- Sentinel-1A/1B POEORB `.EOF`: XML Orbit State Vectors с полями
  `X, Y, Z, VX, VY, VZ`.

GNSS CODE MGEX `COD0MGXFIN_20240070000_01D_05M_ORB.SP3.gz` оставлен в
ноутбуках отдельно как `POSITION_ONLY_GNSS_SOURCES`. Он полезен для широкой
проверки MEO/GEO, но не входит в основной velocity-debug сценарий, потому что в
этом SP3-продукте нет `V`-строк.

В проекте это вынесено в два модуля:

- `dynamics.sp3`: читает `.SP3`, `.SP3.gz` и `.ZIP` с SP3-файлом. Позиции SP3
  переводятся из километров в метры, velocity records - из дециметров в секунду
  в метры в секунду;
- `dynamics.eof`: читает Sentinel EOF/XML. Позиции EOF уже заданы в метрах,
  скорости - в метрах в секунду, эпохи - UTC.

Функции `sp3_state_samples` и `eof_state_samples` возвращают одинаковый
контракт: `Time` и массив состояний GCRS `[x, y, z, vx, vy, vz]` в единицах
`[м, м/с]`. Для источников с `requires_velocity=True` ноутбуки явно проверяют
наличие штатных скоростей и не переходят молча к конечным разностям.

### Приближенные параметры аппаратов

`SpacecraftProperties` в демо-ноутбуках задаются как воспроизводимая первая
оценка, а не как результат идентификации баллистического коэффициента. Массы
взяты из открытых описаний миссий, а `reference_area` является одной
характерной эффективной площадью, используемой текущей простой моделью и для
drag, и для SRP:

| Аппарат | `mass` [кг] | `reference_area` [м^2] | Основание оценки |
| --- | ---: | ---: | --- |
| Sentinel-1A/B | 2185 | 10.3 | ESA указывает массу 2185 кг; характерная площадь взята по размеру C-SAR антенны 12.3 x 0.84 м |
| CASSIOPE / Swarm-E | 500 | 2.2 | e-POP указывает 500 кг и hexagonal bus 1.8 м across x 1.25 м high; площадь оценена как порядок боковой/лобовой проекции |
| Swarm A/B/C | 438 | 1.1 | Для Swarm orbit insertion опубликованы mass after commissioning 438 кг и average nominal drag area 1.1 м^2 |

Во всех этих строках используются `Cd = 2.2` и `Cr = 1.3`. Для Swarm эти
значения согласованы с orbit-insertion документом; для Sentinel и CASSIOPE это
номинальные инженерные коэффициенты, которые стоит уточнять или оценивать по
остаточным невязкам. Текущая модель не учитывает ориентацию, панельную
геометрию и разные площади для drag и SRP.

### Центральное поле Земли

Базовое гравитационное ускорение считается как поле сферической Земли:

```text
a_earth = -mu_E * r / |r|^3
```

где `mu_E = 3.986004418e14 м^3/с^2`. Эта часть задает основное кеплерово
движение вокруг Земли.

### Поправка J2

Сплюснутость Земли учитывается первой зональной гармоникой `J2`:

```text
a_J2,x = k x (5 z^2 / r^2 - 1)
a_J2,y = k y (5 z^2 / r^2 - 1)
a_J2,z = k z (5 z^2 / r^2 - 3)

k = 3/2 * J2 * mu_E * R_E^2 / r^5
r = |r|
```

Используемые константы: `J2 = 1.08262668e-3`, `R_E = 6378136.3 м`.
Эта поправка в первую очередь нужна для воспроизведения вековой прецессии
линии узлов и аргумента перицентра.

Через `ForceModelConfig.j2_frame` доступны два варианта расчета:

- `j2_frame="gcrs_fixed_axis"` - старый воспроизводимый режим, где формула выше
  применяется прямо в координатах состояния;
- `j2_frame="itrs_body_fixed"` - положение спутника поворачивается из GCRS в
  земную ITRS, J2 считается относительно оси вращения твердой Земли, затем
  вектор ускорения поворачивается обратно в GCRS.

Схема body-fixed режима:

```text
r_GCRS(t) -> R_GCRS_to_ITRS(t) r_GCRS(t) = r_ITRS(t)
r_ITRS(t) -> a_J2_ITRS(t)
a_J2_GCRS(t) = R_GCRS_to_ITRS(t)^T a_J2_ITRS(t)
```

Поскольку интегрирование остается в GCRS, кориолисовы и центробежные члены
здесь не добавляются: вращается только физический вектор ускорения. Матрица
`R_GCRS_to_ITRS(t)` строится средствами Astropy как композиция
`GCRS -> CIRS -> ITRS` и кэшируется для повторных эпох. Для одной осесимметричной
J2-поправки отличие от старого режима обычно мало, но `itrs_body_fixed` является
правильной основой для будущих `C_nm/S_nm` гармоник.

### Статические сферические гармоники EGM2008

Для addition-study можно заменить аналитическую J2-ветку на статическое поле
сферических гармоник:

```python
from dynamics import download_egm2008_gfc

download_egm2008_gfc("data/cache/egm2008.gfc")

cfg.environment.force_models.earth_gravity_model = "egm2008"
cfg.environment.force_models.gravity_max_degree = 8
cfg.environment.force_models.gravity_max_order = 8
cfg.environment.gravity_coefficients_file = "data/cache/egm2008.gfc"
```

Коэффициенты читаются из локального ICGEM `.gfc` файла функцией
`read_icgem_gfc(...)`. Большие файлы EGM2008 не коммитятся в репозиторий:
путь к ним должен быть сохранен в параметрах эксперимента. Расчет выполняется
в body-fixed ITRS, а гармоническое ускорение поворачивается обратно в GCRS.

В первой версии ускорение от гармонического потенциала считается численным
градиентом. Это проще проверять и уже согласовано с аналитической J2 при
`C20 = -J2 / sqrt(5)`, но медленнее аналитических производных. Поэтому для
ноутбуков стоит начинать с малых `gravity_max_degree/gravity_max_order`, например
8 или 16, и отдельно фиксировать выбранную степень в summary-таблицах.

### Притяжение Солнца и Луны

Солнце и Луна учитываются как третьи тела в геоцентрической системе. Для тела
с гравитационным параметром `mu_b` и геоцентрическим положением `r_b`:

```text
a_body = mu_b * ((r_b - r) / |r_b - r|^3 - r_b / |r_b|^3)
```

Вторая часть формулы вычитает ускорение центра Земли тем же телом, поэтому
в модели остается именно дифференциальное возмущение орбиты спутника. Положения
Солнца и Луны берутся из `astropy.coordinates.get_body_barycentric`.

### Аэродинамическое сопротивление

Сопротивление атмосферы считается по баллистической модели:

```text
a_drag = -1/2 * Cd * A / m * rho * |v_rel| * v_rel
v_rel = v - omega_E x r
```

где `Cd` - коэффициент сопротивления, `A` - эффективная площадь [м^2], `m` -
масса [кг], `rho` - плотность атмосферы [кг/м^3], `omega_E` - угловая скорость
вращения Земли. Скорость берется относительно атмосферы, которая приближенно
считается совместно вращающейся с Землей.

Модель плотности выбирается через `ForceModelConfig.density_model`:

- `"nrlmsise00"` - использовать NRLMSISE-00, если установлен одноименный
  опциональный пакет;
- `"exponential"` - использовать простую экспоненту без обращения к Astropy и
  NRLMSISE-00.

Старый флаг `nrlmsise00_atmosphere=False` сохраняется для совместимости и
принудительно переводит расчет на экспоненту. Для NRLMSISE-00 положение
спутника преобразуется из GCRS в ITRS средствами Astropy, затем в модель
передаются широта, долгота и высота. При отсутствии или ошибке `nrlmsise00`
используется резервная экспоненциальная модель:

```text
rho(h) = rho0 * exp(-(h - h0) / H)
rho0 = 1.225 кг/м^3
h0 = 0 м
H = 8500 м
```

Эта fallback-модель проста и воспроизводима, но физически груба для орбитальных
высот.

Для воспроизводимых исторических запусков NRLMSISE-00 может брать индексы
солнечной и геомагнитной активности из CelesTrak Space Weather CSV. Файл можно
скачать функцией:

```python
from dynamics import download_celestrak_space_weather_csv

download_celestrak_space_weather_csv("data/cache/SW-All.csv")
```

Затем путь передается в `EnvironmentConfig.space_weather_file`. Парсер читает
`F10.7_ADJ`, `F10.7_ADJ_CENTER81`, `AP1..AP8` и `AP_AVG`; для текущей UTC-эпохи
выбирается соответствующий 3-часовой `AP`-интервал. Если файл не задан, дата
отсутствует в таблице или строка неполная, используется явный quiet fallback:
`f107a = 150`, `f107 = 150`, `ap = 4`.

### Давление солнечного излучения

Давление солнечного излучения считается в cannonball-приближении:

```text
a_srp = nu * P0 * (AU / d)^2 * Cr * A / m * e_sun_to_sc
d = |r - r_sun|
e_sun_to_sc = (r - r_sun) / d
```

где `P0 = 4.56e-6 Н/м^2` - давление на 1 а.е., `AU = 149597870700 м`, `Cr` -
коэффициент отражения, `nu` - доля видимого солнечного диска. Направление
ускорения задается от Солнца к спутнику.

Через `ForceModelConfig.srp_shadow_model` можно выбрать модель тени Земли:

- `"none"` - старый режим без затенения, `nu = 1`;
- `"cylindrical"` - бесконечный цилиндр за Землей, `nu` принимает только 0 или 1;
- `"conical"` - Солнце и Земля считаются угловыми дисками, а `nu` вычисляется
  как неперекрытая доля солнечного диска. Это дает плавную полутень.

В текущей версии Земля и Солнце считаются сферами с радиусами `R_E = 6378136.3 м`
и `R_S = 695700000 м`; рефракция атмосферы и форма Земли в eclipse-модели не
учитываются.

### Тепловое ИК-излучение Земли

Для addition-study добавлена простая модель давления собственного теплового
излучения Земли:

```text
a_ir = F_ir / c * (R_E / r)^2 * Cr * A / m * r / |r|
```

где `F_ir = 239 Вт/м^2` - средний outgoing longwave radiation у Земли,
`c = 299792458 м/с`, `r = |r|`. Модель включается через
`ForceModelConfig.earth_radiation_model = "isotropic_ir"` и по умолчанию
выключена (`"none"`). Это изотропное сферическое приближение: альбедо,
широтно-долготная карта излучения, облачность, сезонность и затенение Землей
солнечного излучения внутри этой силы не моделируются.

### Релятивистская поправка

Через `ForceModelConfig.relativity_model = "schwarzschild"` можно включить
первую постньютоновскую поправку к центральному полю Земли:

```text
a_rel = mu_E / (c^2 r^3) * ((4 mu_E / r - v^2) r_vec + 4 (r_vec · v_vec) v_vec)
```

где `c = 299792458 м/с`. По умолчанию `relativity_model = "none"`. Для LEO эта
поправка мала по сравнению с J2 и сопротивлением атмосферы, поэтому она
предназначена прежде всего для отдельного addition-сравнения и контроля
физической полноты модели.

### Приливная поправка твердой Земли

Прямое притяжение Солнца и Луны уже учитывается как third-body acceleration.
Дополнительно можно включить деформацию гравитационного поля Земли от solid
Earth tides:

```text
Delta U_b = k2 * mu_b * R_E^5 / r_b^3 / r^3 * P2(cos psi)
P2(x) = (3 x^2 - 1) / 2
a_tide = grad(Delta U_moon + Delta U_sun)
```

Расчет включается через `ForceModelConfig.tide_model = "solid_earth_degree2"`.
В текущей реализации используется только degree-2 упругая приливная поправка от
Луны и Солнца с фиксированным Love number `k2 = 0.3`. Вектор спутника и векторы
тел сначала поворачиваются в ITRS, приливное ускорение считается в земной
системе, затем поворачивается обратно в GCRS. По умолчанию
`tide_model = "none"`.

Это не полная IERS 2010 модель: не учитываются частотная зависимость Love
numbers, ocean tides, pole tide, permanent tide conventions и локальные loading
эффекты.

## Численное интегрирование

`propagate_orbit` поддерживает два внутренних интегратора:

- `integrator="rk4_fixed"` - воспроизводимый RK4 с постоянным шагом вывода
  `step_seconds`;
- `integrator="dop853"` - адаптивный `scipy.integrate.solve_ivp` с
  `method="DOP853"`.

Историческое значение `integrator="DOP853"` принимается как алиас к
`"dop853"`, чтобы старые ноутбуки оставались запускаемыми. Параметр `backend`
пока оставлен как интерфейсная точка для будущего сравнения с TudatPy или
Orekit, но не меняет реализацию.

Для DOP853 `step_seconds` задает только сетку сохранения результата `t_eval`.
Внутренний шаг выбирается адаптивно по относительному допуску `rtol` и
абсолютному допуску `atol`. Рекомендуемый начальный вектор:

```python
atol = np.array([1e-3, 1e-3, 1e-3, 1e-6, 1e-6, 1e-6])
```

Первые три компоненты относятся к координатам `[м]` и соответствуют 1 мм,
последние три относятся к скоростям `[м/с]` и соответствуют 1 мкм/с. Эти
допуски ограничивают локальную численную ошибку интегратора, но не являются
оценкой физической точности модели.

Для воспроизводимости рекомендуется явно сохранять:

- начальное состояние `initial_state`;
- UNIX-эпоху `epoch_seconds`;
- длительность и шаг интегрирования;
- интегратор, `rtol` и `atol`;
- параметры `SpacecraftProperties`;
- значения флагов `ForceModelConfig`.

## Сохранение baseline/addition результатов

Для сравнения вариантов модели добавлен модуль `dynamics.experiment_results`.
Он считает стандартные residual-метрики по уже выровненным массивам состояний
`[x, y, z, vx, vy, vz]` в `[м, м/с]`:

- `|dr|` - полная позиционная невязка [км];
- `|dh|` - невязка высоты [км];
- `|dv|` - скоростная невязка [км/с].

Функция `write_experiment_summary(...)` сохраняет результаты в JSON и CSV.
JSON содержит источник reference-данных, параметры `PropagationConfig`,
`SpacecraftProperties`, `ForceModelConfig` и summary-метрики; CSV оставляет
компактную таблицу медианных и максимальных ошибок. В
`prediction_error_growth_demo.ipynb` нижняя ячейка записывает файлы в
`history/prediction_error_growth_addition/`.

Для однотипных addition-прогонов есть `make_addition_force_configs()`. Он
возвращает именованные конфигурации:

- `recommended_baseline`;
- `plus_earth_ir`;
- `plus_relativity`;
- `plus_solid_earth_tides`.

В `prediction_error_growth_demo.ipynb` также есть optional-сценарий
`plus_egm2008_degree8`; он появляется только если локально существует
`data/cache/egm2008.gfc`.

## Ограничения модели

- Форма и ориентация аппарата не моделируются: используются постоянные `Cd`,
  `Cr`, `A` и `m`.
- Затенение Землей для давления солнечного излучения доступно как
  `srp_shadow_model`, но по умолчанию выключено для совместимости со старыми
  baseline-запусками.
- Давление теплового ИК-излучения Земли реализовано только в изотропном
  сферическом приближении и по умолчанию выключено.
- Релятивистская поправка реализована только как Schwarzschild-член для
  центрального поля Земли; эффекты вращения Земли и более полная ОТО-модель не
  учитываются.
- Приливная поправка ограничена degree-2 solid Earth tide; океанические приливы
  и частотные поправки пока не реализованы.
- EGM2008-подобная модель требует локальный ICGEM `.gfc` файл и в первой версии
  использует численный градиент потенциала, поэтому высокие степени могут быть
  медленными.
- Атмосфера считается совместно вращающейся с Землей без ветров.
- В NRLMSISE-00 используются фиксированные спокойные индексы солнечной и
  геомагнитной активности только если не задан `space_weather_file` или в нем
  нет подходящей строки.
- Для fallback-плотности используется одна экспоненциальная шкала высоты,
  поэтому она не должна рассматриваться как эталонная модель атмосферы.
- Для RK4 `step_seconds` является реальным шагом интегрирования; для DOP853 это
  только шаг сохранения результата.
- Основная валидация использует LEO precise-orbit источники со штатными скоростями: Swarm, CASSIOPE/Swarm-E и Sentinel-1. GNSS CODE MGEX оставлен как optional position-only проверка средних/высоких орбит.
- Системы времени, преобразования координат и эфемериды не реализуются вручную,
  а делегируются Astropy.

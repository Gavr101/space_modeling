# План уточнения модели полета спутников

## 1. Краткий анализ исходного плана

Исходный план my_upg_plan.md правильно выделяет пять основных направлений: поле тяготения
Земли, атмосферное сопротивление, численное интегрирование, давление излучения
и релятивистские поправки. Для текущего состояния проекта важны две вещи.

Во-первых, в коде уже есть удобная основа для addition/ablation-исследований:
`ForceModelConfig` включает и отключает отдельные силы, а ноутбуки уже умеют
сравнивать результат с precise-orbit источниками со штатными скоростями. Эту
архитектуру нужно сохранить.

Во-вторых, часть текущих флагов пока грубее, чем выглядит снаружи:

* `PropagationConfig.integrator = "DOP853"` сейчас не используется как реальный
  выбор метода; фактически всегда работает RK4 с постоянным шагом.
* `J2` считается прямо в координатах состояния, то есть как поправка с
  фиксированной осью `z` в GCRS. Для одной осесимметричной J2-поправки суточное
  вращение Земли вокруг этой оси почти не важно, но для строгой модели и тем
  более для `C_nm/S_nm` гармоник выше второго порядка расчет должен идти в
  системе, связанной с Землей.
* NRLMSISE-00 уже предусмотрена как опциональная модель, но солнечные и
  геомагнитные индексы пока зафиксированы константами `f107a = 150`,
  `f107 = 150`, `ap = 4`.
* Давление солнечного излучения уже есть, но без тени Земли.
* Релятивистских поправок пока нет.

Главный принцип внедрения: каждое изменение должно быть переключаемым, иметь
старый вариант для сравнения и проверяться на тех же reference-спутниках
Swarm/CASSIOPE/Sentinel. Для сохранения хорошей предсказательной способности
основной сценарий проверки должен быть addition-study: берем лучший
воспроизводимый baseline и добавляем новые силы по одной, оставляя в
рекомендуемой конфигурации только те компоненты, которые устойчиво улучшают
прогноз.

## 2. Приоритетная последовательность работ

### Этап 0. Зафиксировать baseline

Перед изменением физики нужно сохранить текущие численные результаты как
контрольную точку.

Что сделать:

* Добавить простой скрипт/ячейку в ноутбуках для сохранения baseline-таблицы:
  источник, эпоха, горизонт прогноза, шаг, параметры КА, включенные силы,
  медианные/максимальные ошибки `|dr|`, `|dh|`, `|dv|`.
* Прогнать текущие `orbit_propagation_demo.ipynb` и
  `prediction_error_growth_demo.ipynb` из корня репозитория и из папки
  `notebooks/`.
* Сохранить baseline отдельно от исходных данных, например в `history/`.

Критерий готовности: после каждого следующего этапа можно сказать не только
"стало красивее", а количественно сравнить ошибки с прежней версией.

### Этап 1. Исправить интерфейс интеграторов

Выбранное решение: использовать уже имеющийся `scipy.integrate.solve_ivp` и
реализовать реальный переключатель между старым RK4 и адаптивным DOP853.

Почему так:

* SciPy уже есть в зависимостях проекта.
* DOP853 в SciPy является явным методом Рунге-Кутты 8-го порядка; документация
  SciPy рекомендует его для задач высокой точности при малых `rtol/atol`.
* Реализация Everhart/IAS15 потенциально интересна, но это отдельная большая
  работа. Ее лучше делать после того, как появится корректный baseline для
  DOP853.

Что сделать:

* Заменить строковое поле `integrator: str` на ограниченный набор значений:
  `"rk4_fixed"` и `"dop853"`.
* Сохранить старый RK4 как `rk4_fixed` без изменения формулы шага.
* Для `"dop853"` вызывать `solve_ivp` с `method="DOP853"`, `t_eval` на той же
  сетке вывода, что и текущий `step_seconds`.
* Добавить в `PropagationConfig` параметры `rtol` и `atol`. Начальное значение
  `rtol = 1e-10`. Для `atol` использовать не одно число, а вектор под разные
  единицы состояния:

```python
atol = np.array([1e-3, 1e-3, 1e-3, 1e-6, 1e-6, 1e-6])
```

  Первые три компоненты задают абсолютный допуск координат `[м]` на уровне
  1 мм, последние три - абсолютный допуск скоростей `[м/с]` на уровне
  1 мкм/с. Это не гарантирует физическую точность модели, а только ограничивает
  локальную численную ошибку адаптивного интегратора.
* В README явно описать, что `step_seconds` для DOP853 задает сетку сохранения,
  а не внутренний шаг интегратора.

Критерий готовности: при `integrator="rk4_fixed"` результаты совпадают с
baseline в пределах машинного шума; при `integrator="dop853"` ноутбуки
выполняются без изменения пользовательского сценария.

### Этап 2. Привести J2 и будущие гармоники к земной системе координат

Выбранное решение для ближайшей реализации: оставить аналитическую J2-формулу,
но считать ее в ITRS/ITRF и возвращать ускорение обратно в GCRS.

Схема:

```text
r_GCRS(t)
  -> R_GCRS_to_ITRS(t)
r_ITRS(t) = R_GCRS_to_ITRS(t) r_GCRS(t)
  -> a_J2_ITRS(r_ITRS)
a_J2_GCRS(t) = R_GCRS_to_ITRS(t)^T a_J2_ITRS(t)
```

Важно: для ускорения не нужно добавлять кориолисовы члены, если интегрирование
идет в квазиинерциальной GCRS. Мы поворачиваем физический вектор силы, а не
переписываем уравнения движения во вращающейся системе.

Что сделать:

* Добавить переключатель модели J2:
  `"gcrs_fixed_axis"` для текущего варианта и `"itrs_body_fixed"` для нового.
* Использовать Astropy для получения преобразования GCRS/ITRS на эпоху `t`.
* Не строить тяжелые `SkyCoord`/`GCRS`-объекты в каждой правой части без кэша:
  для первой версии допустим прямой Astropy-вызов, но сразу добавить простой
  кэш матриц поворота по времени RHS или по сетке интегрирования.
* Проверить, что старый и новый J2 отличаются мало на коротком LEO-интервале,
  но новый вариант является правильной основой для гармоник `C_nm/S_nm`.

Критерий готовности: J2 можно независимо переключать между старой и новой
реализацией, а тесты подтверждают размерности, единицы и отсутствие скачков
ускорения при переходе через сутки.

### Этап 3. Добавить усеченную модель гравитационного поля EGM2008

Выбранное решение: реализовать собственный расчет ускорения по сферическим
гармоникам в body-fixed системе, а коэффициенты брать из открытого EGM2008.
Не подключать тяжелый astrodynamics-backend ради одной силы.

Почему так:

* Ядро пропагации по правилам проекта должно оставаться собственным.
* NGA публикует EGM2008 spherical harmonics как открытый набор коэффициентов.
* Полный EGM2008 слишком большой для быстрого демо, поэтому нужен
  настраиваемый `max_degree/max_order`, например 8, 16, 32.

Что сделать:

* Добавить новую модель `earth_gravity_model = "j2" | "egm2008"`.
* Для `"egm2008"` читать коэффициенты из локального файла или из кэша загрузки;
  исходные большие файлы не коммитить в репозиторий.
* По умолчанию использовать малую степень, например `max_degree=8`, и явно
  записывать ее в результаты экспериментов.
* Считать гармоники в ITRS, затем поворачивать ускорение в GCRS той же
  инфраструктурой, что и для body-fixed J2.
* Сначала реализовать только статическое поле без приливов.

Что не делать в первой итерации:

* Не внедрять приливы одновременно с EGM2008. Твердые/океанические приливы и
  IERS 2010 нужны для высокоточной POD-модели, но они сильно усложняют
  валидацию. Их лучше добавить после того, как статические гармоники покажут
  измеримое улучшение.

Критерий готовности: для `max_degree=2` новая гармоническая модель согласуется
с J2-тестом, а для `max_degree > 2` addition/ablation-сравнение показывает
влияние на Swarm, CASSIOPE и Sentinel.

### Этап 4. Улучшить атмосферу: NRLMSISE-00 + воспроизводимые индексы

Выбранное решение: сделать NRLMSISE-00 основной высокоточной атмосферной
моделью, но оставить экспоненту как fallback и baseline.

Почему так:

* NRLMSISE-00 специально описывает температуру, состав и массовую плотность
  верхней атмосферы до экзосферы.
* ГОСТ Р 25645.166-2004 может быть полезен для сравнения, но для Python-кода
  проще и воспроизводимее сначала довести до ума уже подключенную NRLMSISE-00.

Что сделать:

* Разделить выбор модели плотности:
  `"exponential"` и `"nrlmsise00"`.
* Оформить `nrlmsise00` как опциональную зависимость/extra, но не делать ее
  обязательной для установки проекта.
* Добавить загрузчик space-weather индексов из CelesTrak Space Weather:
  `F10.7`, 81-дневный `F10.7a`, `Ap`/3-hour Ap.
* Кэшировать загруженные индексы в `data/cache/` или аналогичной папке,
  сохранять URL, дату загрузки и диапазон дат.
* Если индексов нет, не падать молча: явно писать в результат эксперимента,
  что использованы спокойные константы.
* Для исторических precise-orbit тестов использовать индексы именно на эпоху
  reference-данных, а не текущую дату.

Критерий готовности: в notebook-выводе видно, какие `f107/f107a/ap`
использованы для каждого запуска; отключение NRLMSISE-00 воспроизводит старую
экспоненциальную ветку.

### Этап 5. Добавить тень Земли для давления солнечного излучения

Выбранное решение: реализовать сферическую коническую модель тени с плавным
коэффициентом освещенности от 0 до 1.

Почему так:

* Текущая SRP-модель всегда действует, даже когда аппарат находится в тени.
* Коническая модель лучше цилиндрической для переходов через полутень и не
  требует сложной модели атмосферы/рефракции.
* Orekit использует явную концепцию umbra/penumbra для eclipse detection; это
  хороший ориентир для интерфейса, даже если сам код остается собственным.

Что сделать:

* Добавить `srp_shadow_model = "none" | "cylindrical" | "conical"`.
* По умолчанию для новой high-fidelity конфигурации использовать `"conical"`.
* Умножать текущее SRP-ускорение на `illumination_factor`.
* Для первой версии считать Землю и Солнце сферами с радиусами `R_EARTH` и
  `R_SUN`; положение Солнца брать тем же способом, что и сейчас.
* Добавить тесты для трех случаев: полное освещение, umbra, penumbra.

Критерий готовности: SRP можно запускать с тенью и без тени, а на графиках
видны интервалы исчезновения SRP-ускорения при входе в тень.

### Этап 6. Добавить давление ИК-излучения Земли

Выбранное решение: добавить простую переключаемую модель Earth radiation
pressure для теплового ИК-излучения Земли, выключенную по умолчанию до
addition-проверки.

Почему так:

* Для LEO-аппаратов вклад ИК-излучения обычно мал по сравнению с drag, но он
  действует регулярно и может проявляться в остатках, особенно когда атмосфера
  уже улучшена.
* Текущая модель аппарата использует одну эффективную площадь и не знает
  ориентации, поэтому начинать нужно с простой cannonball-модели, а не с
  панельной тепловой модели.
* Добавление отдельным флагом позволяет проверить влияние без риска ухудшить
  основной прогноз.

Что сделать:

* Добавить переключатель `earth_ir_pressure: bool = False` или
  `earth_radiation_model = "none" | "isotropic_ir"`.
* В первой версии считать Землю сферическим изотропным источником теплового
  излучения. Ускорение направлять от центра Земли по `r_hat`, модуль считать
  как давление излучения на расстоянии спутника, умноженное на
  `Cr_earth * A / m`.
* Ввести явные константы с единицами: эффективный поток ИК-излучения Земли
  `[Вт/м^2]`, скорость света `[м/с]`, радиус Земли `[м]`.
* Не смешивать ИК-излучение с альбедо в первой реализации. Альбедо добавить
  позже отдельным режимом `earth_radiation_model = "ir_albedo"` только если
  простая ИК-модель дает измеримый вклад или хорошо объясняет residuals.
* В диагностике выводить порядок величины `|a_earth_ir|` на первом шаге и его
  отношение к `|a_srp|` и `|a_drag|`.

Критерий готовности: модель можно включить одной настройкой, она не меняет
результаты при выключенном флаге, а addition-таблица показывает отдельный
эффект `baseline -> +earth_ir_pressure`.

### Этап 7. Добавить приливные поправки к гравитационному полю

Выбранное решение: добавить приливы как time-variable поправки к земному
потенциалу, а не как замену уже имеющихся третьих тел. Притяжение Солнца и
Луны как третьих тел в коде уже есть; новый этап должен описывать деформацию
гравитационного поля Земли под действием Луны и Солнца.

Почему так:

* Для высокоточной орбитальной динамики приливы входят через временные
  изменения гармонических коэффициентов гравитационного поля Земли.
* Твердые земные приливы можно реализовать первым приближением относительно
  компактно через низкие степени и Love numbers.
* Океанические приливы потенциально важны, но требуют внешних таблиц
  коэффициентов и аккуратного учета воспроизводимости.

Что сделать:

* Добавить переключатель `tide_model = "none" | "solid_earth_degree2"`.
* В первой версии реализовать только твердые земные приливы степени 2 от Луны
  и Солнца как поправки к `C20/C21/S21/C22/S22` в body-fixed системе.
* Использовать положения Луны и Солнца из уже подключенной Astropy-инфраструктуры
  и ту же матрицу GCRS/ITRS, что для EGM/J2.
* Явно зафиксировать Love number `k2` и источник константы в README/комментарии.
* Океанические приливы вынести во вторую итерацию:
  `tide_model = "solid_earth_degree2_ocean"` или отдельный флаг
  `ocean_tides`, после выбора конкретного набора коэффициентов и формата кэша.
* В addition-таблице проверять отдельно:
  `baseline -> +solid_earth_tides`, а затем, если будет реализовано,
  `+ocean_tides`.

Критерий готовности: при `tide_model="none"` результаты совпадают с baseline,
при включении твердых приливов ускорение имеет разумный порядок величины, а
сравнение на нескольких спутниках показывает, улучшает или ухудшает эта сила
прогноз.

### Этап 8. Добавить релятивистскую поправку как дешевый optional-force

Выбранное решение: добавить первую постньютоновскую поправку Шварцшильда от
центрального тела Земли и держать ее отдельным флагом.

Ориентировочная формула:

```text
a_rel = mu_E / (c^2 r^3) * ((4 mu_E / r - v^2) r_vec + 4 (r_vec · v_vec) v_vec)
```

Что сделать:

* Добавить `relativity_correction: bool = False` или отдельный выбор модели
  `"none" | "schwarzschild"`.
* Включать ее только в addition-наборах, пока не будет видно устойчивого
  улучшения на precise-orbit данных.
* Документировать, что это не полная релятивистская модель IERS для всех тел,
  а первый практический шаг.

Критерий готовности: поправка мала по модулю, не ломает устойчивость
интегрирования, ее вклад виден отдельно в диагностической таблице ускорений.

## 3. Изменения публичных интерфейсов

Минимальный набор изменений должен выглядеть так:

* `ForceModelConfig` сохраняет старые bool-флаги, но получает параметры выбора:
  `earth_gravity_model`, `j2_frame`, `density_model`, `srp_shadow_model`,
  `earth_radiation_model`, `tide_model`, `relativity_model`.
* `PropagationConfig` получает реальные настройки интегратора:
  `integrator`, `rtol`, `atol`, где `atol` для шестимерного состояния по
  умолчанию задается вектором `[м, м, м, м/с, м/с, м/с]`.
* Старые режимы остаются доступными для ablation:
  `rk4_fixed`, `gcrs_fixed_axis`, `exponential`, `srp_shadow_model="none"`,
  `earth_radiation_model="none"`, `tide_model="none"`.
* Новая рекомендуемая конфигурация для ноутбуков явно задается в одном helper,
  например `make_recommended_force_config()`, чтобы не размазывать настройки
  по ячейкам.
* Все новые публичные функции получают аннотации типов и docstring с единицами
  измерения.

## 4. Проверка и критерии принятия

Обязательные проверки после каждого этапа:

* `python -m compileall dynamics`
* `python -m pytest`, если окружение позволяет писать pytest cache.
* Синтаксическая проверка code cells обоих ноутбуков.
* Выполнение `orbit_propagation_demo.ipynb` и
  `prediction_error_growth_demo.ipynb` из корня репозитория.
* Выполнение тех же ноутбуков из рабочей директории `notebooks/`.

Научные проверки:

* Проверять единицы измерения всех новых ускорений: итог всегда `[м/с^2]`.
* Для каждой силы выводить хотя бы диагностический порядок величины ускорения
  на первом шаге.
* Для каждой новой силы делать addition-проверку: текущий лучший baseline,
  baseline + одна новая сила, разность ошибок по `|dr|`, `|dh|`, `|dv|`.
  Отдельные ablation-прогоны использовать позже, когда появится собранная
  high-fidelity конфигурация.
* Сортировать сравнение спутников по фактической медианной высоте, как уже
  сделано в новых ноутбуках.
* Считать улучшением только устойчивое уменьшение ошибок на нескольких
  аппаратах, а не единичный выигрыш на одном источнике.

## 5. Риски и допущения

* Astropy-преобразования GCRS/ITRS физически удобны, но слишком дороги для
  вызова без кэша на каждой правой части ОДУ. Поэтому после корректной первой
  реализации нужен кэш ориентации или предрасчет матриц.
* EGM2008 в высокой степени может дать выигрыш только при достаточно точных
  начальных условиях, корректной системе координат и хорошей атмосфере. Если
  drag-модель грубая, выигрыш от высоких гармоник может потеряться в шуме.
* Негравитационные силы зависят от ориентации и площади аппарата. Текущая
  `SpacecraftProperties.reference_area` является грубой эффективной площадью,
  поэтому нельзя ожидать POD-точности без оценки параметров.
* Space-weather индексы должны соответствовать эпохе reference-данных; текущие
  индексы для "сегодня" не подходят для исторических SP3/EOF файлов.
* ИК-излучение Земли и приливы должны быть выключены по умолчанию до
  addition-проверки. Их нельзя автоматически добавлять в рекомендуемую модель,
  если они улучшают только один спутник и ухудшают остальные.
* Полноценные релятивистские модели IERS следует включать только после того,
  как более простые поправки дадут воспроизводимый baseline.
